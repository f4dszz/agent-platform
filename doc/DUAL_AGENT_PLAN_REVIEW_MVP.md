# 双 Agent 计划-审核 MVP 说明

## 1. 背景

当前方向不是去做一个通用任务编排平台，也不是单纯做一个好看的聊天室。

真实痛点是：当一个复杂任务需要两个原生 agent 从不同视角协作时，手工方式很低效。典型例子是项目重构：一个 agent 先出计划，另一个 agent 再审核。现在通常只能开两个终端，把第一个的输出复制粘贴给第二个，再人工来回传话，这很麻烦，也容易丢上下文。

所以这个项目更合理的定义是：

以房间为展示层的原生 agent 协作工作台。

房间保留“像数字员工协作”的体验，但底层真正解决的是会话移交、审核、修订和人工决策。

## 2. 核心判断

这个方向本身没有问题，但要收敛。

正确的第一步不是做“很多 agent 自由聊天”，而是先做清晰的双 agent 工作流：

- Agent A 负责产出计划
- Agent B 负责审核计划
- 人负责最终决策

这样做的价值是真实的，因为它直接替代了“两个终端 + 手工复制粘贴”的低效操作。

## 3. 这条路和通用任务编排平台的区别

市面上的 Coze、Dify、LangChain、AutoGen 主要是把模型、工具、工作流组合成一套可配置执行系统。

本项目的重点不同：

- 优先接入原生 agent，而不是自己重做 agent 内核
- 以 room 为核心，而不是以 workflow builder 为核心
- 强调长期协作空间，而不是一次性执行链路
- 先解决“原生 agent 之间如何顺畅接力和审核”，而不是先追求通用编排能力

但也要明确一点：前台不叫“任务编排平台”，不代表后台不需要最小编排能力。只要涉及计划、审核、修订、定时触发、主动汇报，就必须有状态和流程控制。

## 4. 重构场景的 before / after

Before：

- 用户开两个命令行
- 给 Agent A 讲需求
- 等 Agent A 输出计划
- 复制计划内容
- 粘贴给 Agent B
- 再人工整理分歧点
- 再决定是否让 Agent A 修订

After：

- 用户在一个 room 内发起“重构计划审核”
- Agent A 输出计划，系统保存为工件
- Agent B 自动收到审核请求，而不是收到一段随手转发的聊天
- 系统展示计划、审核意见、分歧点
- 用户决定采纳、驳回或要求修订

这里真正提升效率的不是“大家都能聊天”，而是“系统替用户完成移交和审核组织”。

## 5. 第一版只做什么

第一版只聚焦一个强场景：项目重构的双 agent 计划-审核。

最小对象：

- Room：协作空间
- Task：本次任务
- Plan：Agent A 的计划结果
- Review：Agent B 的审核结果
- Decision：人的最终决定

最小流程：

1. 用户创建任务，并指定目标目录或相关文件
2. Agent A 输出重构计划
3. 系统把计划固定为一个可审查对象
4. Agent B 基于原始任务 + 计划 + 文件范围做审核
5. 系统展示审核意见和关键分歧
6. 用户决定通过、退回修订或停止

最小界面：

- 左侧仍然是房间列表
- 中间主区域不再只是消息流，还要有任务卡片和结果卡片
- 聊天记录保留，但只是过程，不是最终结果载体

## 6. 第一版明确不做什么

- 不做通用 workflow builder
- 不做多 agent 自由辩论
- 不做复杂自动调度
- 不做“完全自治数字员工团队”
- 不把定时任务和主动汇报放到第一优先级

这些都可以以后做，但不是当前最该验证的价值。

## 7. 为什么这样收缩

原因很简单：

- 价值更真实，能直接替代手工操作
- 结果更容易判断，能比较“单 agent”与“双 agent 审核”是否更好
- 产品边界更清晰，不会一开始掉进大而全平台竞争
- 工程复杂度可控，适合当前项目阶段

## 8. 当前项目的下一步

不要继续围绕“聊天室能不能更像团队”发散。

下一步应该围绕下面三件事实现：

- 把一次任务从“消息”升级成“可追踪的计划-审核流程”
- 把 Agent 输出从“聊天文本”升级成“计划和审核工件”
- 把人从“手工传话者”升级成“最终决策者”

如果这一版能在真实重构任务里替代两个终端之间的复制粘贴，这个项目就开始真正有用了。

## 9. 实施评审（Claude Code 批注）

### 方向判断

同意。收敛到"双 agent 计划-审核"是对的，不做通用编排也是对的。

### 对第 5 节的意见

第 5 节设计了 Task / Plan / Review / Decision 四个新模型 + 任务卡片 UI。方向正确但步子太大。

当前代码连"agent 能看到聊天室历史"都还没实现。直接跳到结构化工件，等于跳过了最基本的验证：**双 agent 看同一个上下文对话，到底比单 agent 好多少？**

### 建议的实施顺序

**先验证再建模，不要反过来。**

第一步（最小验证，不加新模型）：
1. 聊天历史注入 — agent 调用时带上房间最近 N 条消息，让 Codex 能看到 Claude 刚说了什么
2. agent-to-agent 触发 — Claude 回复后，用户一句 `@codex review above` 就够了，不需要自动触发

跑一次真实的重构任务。如果双 agent 审核确实比单 agent 好，再做第 5 节的结构化。如果没明显提升，那些模型白设计了。

第二步（验证通过后）：
- 加 Task 模型，把"一次计划-审核"从消息流中提取出来
- 加结果对比视图
- 加自动触发（Claude 输出计划后自动 @codex 审核）

### 风险提示

- 聊天历史拼进 prompt 会消耗 token，需要控制条数（建议最近 20 条，约 4000 token）
- Claude `-p` 单次调用的上下文窗口够大，不是瓶颈
- Codex `exec` 的上下文处理能力弱于 Claude，长历史可能降低审核质量
